# 后台定时运行 · 指南

> **一句话**：让一个 AI 引擎在你不在的时候，自动替世界做三件事——往前推、定期体检、定期把流水账提炼成文学。你睡一觉醒来，世界自己走了一夜，回来一刷面板就看见它去哪了。

这套系统的卖点之一是**无人值守长跑**：世界不需要你盯着，它能自行其是地活下去。你手动驱动时，是一次次对引擎说「推进」；**定时任务，就是把"说推进"这件事交给一个闹钟。**

---

## 能托管给后台的三件事

| 代号 | 任务 | 它干什么 | 推荐频率 |
|---|---|---|---|
| **A** | 后台自动推进 | 让世界往前走：逐角色判定、掷骰、写日志 | 每小时一次 **或** 每天一次（步长你自定） |
| **B** | 后台自动体检 | 查世界健康：状态/线索/记忆/膨胀，出一张「建议清单」 | 每周一次 |
| **C** | 后台自动重绘编年史 | 把累积的记录，重写成一版有结构、有文学性的回望 | 每周一次 |

> A 是引擎，B 和 C 是保养。只挂 A 世界也能跑；B、C 让它跑得久了不发胖、不变成流水账。

---

## 开跑前提（三条，第 1、2 条必读）

1. **世界得先创世**。空模板（`当前周期: 0`）挂上后台也不会自己开张——后台只会在 `_log` 记一行「未创世，跳过」。**先手动跟引擎说「创世」、跑顺几次「推进」，确认一切正常，再挂后台。**

2. **搞清你的定时任务是「本地跑」还是「云端跑」**——这是最容易踩的坑：
   - **本地跑**（在你这台电脑上）：直接对着这个 Obsidian 文件夹读写，**最省心，推荐**。
   - **云端跑**（云端 routine / cloud agent）：它在云服务器上，**碰不到你本地的文件夹**。要用云端，得先把仓库推到 GitHub（就是模板待办里的「D·git」那步），让云端 agent 在云上 clone 一份来跑、再把改动同步回来——多一层设置。
   - 拿不准？没关系——下面的「让 AI 帮你建」提示词会**逼 AI 主动告诉你**它建的是哪种。

3. **引擎要求**：任何能①把 `CLAUDE.md` 当操作宪法读、②读写本仓库文件、③跑 `python3` 的环境都行（Claude Code / Codex / Obsidian 里的 Claudian 插件 / Cowork 等）。

## 怎么建任务：两种方式

### 方式一（推荐 · 小白首选）：让 AI 帮你建

你不用懂任何技术细节，**直接把任务交给你的 AI 助手，让它替你建定时器**。Claude 客户端的 cowork、Codex 都有"定时任务 / routine / scheduled task"功能，**AI 知道自己平台怎么建**，你只要把"频率 + 工作区 + 要跑的指令"告诉它。

把下面这段话发给你的 Claude / Codex（以"每小时推进"为例）：

```text
帮我在【这个仓库文件夹】上建一个定时任务（routine / scheduled task）：

- 工作目录 / 工作区：就设成当前这个「故事岛」仓库文件夹。
- 运行频率：每小时一次（cron：0 * * * *）。
- 每次触发时要执行的指令，就是下面这一整段——

〔把本指南后面【A · 推进】那段「任务提示词」整段复制粘贴到这里〕

建好后请明确告诉我三件事：
1. 怎么查看这个任务、怎么暂停或删除它；
2. 它是在我本地电脑上跑，还是在云端跑；
3. 如果是云端跑，提醒我：仓库需要先能在云端访问（先做 git 推 GitHub），否则它碰不到我本地的文件。
```

**要建 B（体检）或 C（重绘编年史）**：把同一段话里的频率换成「每周一次」、把粘贴的指令换成对应的【B】或【C】任务提示词即可。

### 方式二：手动建

在你的 app 的「定时任务 / scheduled task / routine」界面里，**填三样东西**：

| 填什么 | 填成 |
|---|---|
| 工作区 / 工作目录 | 这个「故事岛」仓库文件夹 |
| 运行时间 / 频率 | 一个 cron 表达式（见下方速查表） |
| 指令 / prompt | 对应的【A】/【B】/【C】任务提示词（整段粘贴） |

> **进阶（懂命令行的人）**：也可以用系统自带的 cron / 计划任务，定时以本仓库为工作目录、用 headless 模式调用 `claude` 或 `codex` 跑提示词。这条路要你自己处理好命令行调用与权限授权，**新手建议走 app 界面或方式一**，别为难自己。

## 三类任务的「任务提示词」（每次定时触发实际跑的指令）

> 下面每段都是**自包含**的——后台触发时不带任何对话记忆，所以提示词第一步永远是"先读仓库"。整段复制，别删步骤。

### A · 后台自动推进

**干什么**：把世界向前推进一个步长。**频率**：每小时一次（紧凑地看一个情境慢慢展开），或每天一次（每天早上回来收获一段当日故事）。

```text
你是「故事岛」这个 Obsidian 仓库的世界引擎，现在是一次无人值守的后台定时推进。

1. 先读根目录 CLAUDE.md，把它当操作宪法，全程严格遵守。
2. 读 _state.md。若世界尚未创世（当前周期为 0 或当前日期为空），不要编造任何内容启动——只在 _log.md 顶部记一行「世界未创世，跳过本次定时推进」，然后结束。
3. 若世界已开张：按宪法 §4.1「推进」，把世界向前推进【1 小时】。先逐角色独立判定、关键成败处掷骰，再让行动碰撞成事件，忠实写进当天日志——不要为了「好看」收敛角色的自主与后果；该沉闷、该残酷、该悬而未决就让它发生。
4. 若某个动作会跨过 [[世界设定]] 里写明的红线：按 §3，后台绝不擅自执行，把它冻结成一条「待裁决」写进 _state，照常推进其余部分。
5. 收尾：跑格式闸 python3 08-脚本/lint_day.py、lint_chr.py、lint_state.py 到全绿；跑 python3 08-脚本/build_panel.py 重建面板；写回 _state（只留下一窗口需要的最小快照）；在 _log.md 顶部留一行痕迹，注明现实时间与世界内时间。
```

**步长怎么调**（改提示词里那个【1 小时】）：
- 紧凑写实 → 「1 小时」「3 小时」「推进到傍晚」
- 每日节奏 → 「过一天」或「24 小时」——**跨过午夜会自动触发「日结」**（生成当日故事 + 角色记忆结算），所以每天回来就有一篇故事。
- 大尺度 → 「过一周」；**史诗世界**用你自己的单位：「1 纪」「过一纪」「100 年」。
- 注：哪怕设的是每小时推进，累计跨过午夜时也会自动日结，不会漏。

### B · 后台自动体检（盘点）

**干什么**：给世界做一次健康检查，出一张「哪里该清理"的**建议清单**——但**它只诊断、不动手**。**频率**：每周一次足够。

```text
你是「故事岛」这个 Obsidian 仓库的世界引擎，现在是一次无人值守的后台定时体检（盘点）。

1. 先读根目录 CLAUDE.md，按宪法 §4.4「盘点」执行。
2. 读 _state.md。若世界尚未创世，只在 _log.md 顶部记一行「未创世，跳过」，然后结束。
3. 若已开张，逐项扫查：角色状态、任务进度、悬而未决的线索（含休眠线复核）、失效的 wikilink、角色记忆是否过载、凝练层是否滞后（该建未建的 evt、落后的编年史天数）、动态事实是否膨胀。
4. 自动维护 _log（控制在最新 10 条、超出的移进 09-归档、每条压实到约 2 行）——这一步属于日常维护，直接做，不算建议。
5. 出一份体检报告。所有「该不该改、该怎么改」的判断，必须明确标注【建议】、留给我本人裁决——后台绝不替我裁决、绝不删任何历史、绝不擅改被珍重的层（世界设定 / 角色重要记忆 / 大事件 / 编年史）。把报告写进 _God/体检报告.md（覆盖上一次即可——它是给我看的，放在 _God 后台、引擎平时不读不会污染世界），并在 _log.md 顶部留一行痕迹。
```

> 体检报告落在 `_God/体检报告.md`（你的后台、人看的）。回来翻一眼那张清单，**想采纳哪条建议，再亲口让引擎去做**——主权始终在你手里。

### C · 后台自动重绘编年史（编年 · 重写模式）

**干什么**：把至今累积的记录，**整篇重写**成一版有宏观结构、有文学性的回望（而不是一天一段地往后摞）。**频率**：每周一次。

> ⚠️「重绘」= **先备份旧版、再整篇重写**：每跑一次，会把现有编年史原样存一份进 `09-归档/`（只增不删），然后重写一版新的。所以归档里会逐周攒下历代编年史——这是特性不是 bug。
> （若你更想要"温和追加"而非"整篇重写"，把提示词里的「§4.5 编年·重写 / 重绘模式」换成「§4.5 编年（追加式凝练）」、并删掉第 3 步的备份重写、改成"把近期日志文学性地凝练、追加进编年史当前卷"即可。）

```text
你是「故事岛」这个 Obsidian 仓库的世界引擎，现在是一次无人值守的后台定时「重绘编年史」。

1. 先读根目录 CLAUDE.md，按宪法 §4.5「编年 · 重写 / 重绘模式」执行。
2. 读 _state.md。若世界尚未创世、或世界还太短（累计不足三个周期、没什么可回望的），只在 _log.md 顶部记一行「内容尚浅，跳过重绘」，然后结束。
3. 先把现有 06-编年史/编年史.md 原样复制备份到 09-归档/（只增不删，绝不覆盖已有的旧备份；备份文件名带上当前周期以便区分）。
4. 再以强文学性、宏观结构重写一版新的编年史：整体完整，但可合理精简细节、概括提炼、微调叙事顺序；按结构性的章节组织（不是按周期流水账）；遵守 [[世界设定]] 的文风，说人话，杜绝 AI 味与机械感；必须明显区别于日志和每日故事。
5. 只重写 06-编年史/编年史.md 这一个文件——不改任何史实、不删任何记录、不动其它对象。更新它 frontmatter 的「更新于」，并在 _log.md 顶部留一行痕迹。
```

## 频率速查（cron 表达式）

cron 五位＝`分 时 日 月 周`（周里 0 或 7＝周日，1＝周一）。常用：

| 想要 | cron |
|---|---|
| 每小时（整点） | `0 * * * *` |
| 每 6 小时 | `0 */6 * * *` |
| 每天早 8 点 | `0 8 * * *` |
| 每周一早 9 点 | `0 9 * * 1` |
| 每周日晚 10 点 | `0 22 * * 0` |

**错开时间，别撞车**：如果你同时挂了"每小时推进"和"每周体检"，让体检 / 重绘落在你大概率不会撞上整点推进的时段（这套任务都会改 `_state`、`_log`，两个任务同一刻抢同一批文件容易出乱子）。一个稳妥排法：推进走整点；体检放周一早 9 点、重绘放周日晚 10 点，彼此岔开。

## 后台的安全边界（它绝不会做的事）

这套机制是按宪法「附录 E · 无人值守」设计的，后台运行时**自缚四条**，你尽可放心睡觉：

- **绝不替你创世**：空世界只跳过、不编造。
- **绝不替你裁决**：只给【建议】，采不采纳是你的事。
- **绝不删历史**：只移动 / 压缩 / 改状态，记录永远在（`09-归档/` + 日志兜底）。
- **红线即刹车**：任何动作要跨过你写明的红线，后台不执行，冻结成「待裁决」等你上线。
- 且**每次运行都在 `_log` 留痕**（现实时间 + 世界内时间）——夜里世界走了多远，一目了然。

## 醒来，怎么看世界走了多远

- **`_log.md`** — 每次后台运行的痕迹，最直接：跑了几次、世界走到第几周期。
- **`07-面板/故事岛-面板.html`** — 双击浏览器开，世界全貌；过夜后回来刷新即最新。
- **`06-编年史/`** — 编年史（宏观回望）+ 每日故事（每天一篇，给你读的）。
- **`_state.md` 的「待裁决」区** — 看看有没有撞红线被冻住、等你拍板的事。
- **`_God/体检报告.md`** — 上一次体检的建议清单。
